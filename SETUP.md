## 1. Subagents — a 3-minute primer

You said you've never used subagents. Read this before you do anything else.

### What they are

A Claude Code **subagent** is an isolated Claude session that your main session spawns to do one focused job. The subagent:

- Starts with an **empty context window**. It does not see your conversation, your prior tool calls, or any other subagent's output.
- Sees only: the prompt you (the parent) hand it, its own system prompt (the markdown body of its `.md` file under `.claude/agents/`), env details, and any skills listed in its frontmatter.
- Does its work autonomously (reads files, runs tools).
- Returns **one final message** to the parent — a summary. Intermediate output stays inside the subagent.

### Why they matter for this project

The core LLM failure mode you asked me to defeat is **lost in the middle** — long-context degradation. The trick is to never put the verbose stuff into your main context in the first place.

- Need to fetch the latest Cloudflare Durable Objects docs? Delegate to `docs-researcher` — its 8,000-token research stays inside it, and you get back a 200-token summary.
- Need to review your diff against a 12-principle constitution + 12 privacy invariants + a forbidden-patterns table? Delegate to `reviewer` — same idea.
- Need to check if your i18n catalogs are consistent across 8 locales? Delegate to `i18n-validator`. Etc.

### How you invoke them

Two ways:

1. **Explicit, by name** (what I recommend until you're comfortable):
   > Use the `architect` subagent on `specs/001-onboarding/spec.md`.
   > Use the `reviewer` subagent on the current diff.

2. **Automatic, by description** (later, once you trust it):
   Claude Code reads each subagent's `description` field and auto-delegates when your request matches. The descriptions in `.claude/agents/*.md` are written to make this work.

### The eight subagents in this project

All read-only by design (subagents cannot show permission prompts; the parent does all writes — this is Claude Code best practice).

| Agent | Run after / when |
|---|---|
| `clarifier` | Fresh spec draft, before `/speckit.plan` |
| `architect` | Spec is clarified, before `/speckit.tasks` |
| `docs-researcher` | About to import any library / use any API you haven't freshly verified |
| `test-strategist` | Designing tests for a feature |
| `reviewer` | Implementation done, before commit |
| `security-auditor` | Diff touches PII / crypto / auth |
| `i18n-validator` | Diff touches user-visible strings |
| `platform-parity` | Diff changes shared types or API contracts |

Open `.claude/agents/README.md` for the longer treatment. Each agent's `.md` file is also worth reading — the body is its system prompt.

### One critical implication

A subagent **cannot see your conversation**. If you say "use the reviewer subagent" without telling it what diff to review, it will guess. Always front-load:

> Use the `reviewer` subagent. The diff to review is what `git diff origin/main...HEAD` returns. The spec it claims to satisfy is `specs/001-onboarding/spec.md`.

## 4. API keys to obtain

After the accounts above, generate these API keys. Keep them in a password manager — never commit them anywhere.

### 4.1 Brave Search API key

1. Sign in at https://brave.com/search/api.
2. Subscribe to the **Free** plan (2,000 queries/month — plenty for development).
3. Go to **API keys** → **Add API Key**, name it `boundless-dev`, save the key.

> You'll paste this into `.mcp.json` (next section).

### 4.2 Cloudflare API token

1. Go to https://dash.cloudflare.com/profile/api-tokens.
2. Click **Create Token** → **Custom token**.
3. Permissions needed (initial set — refine later):
   - Account → Workers Scripts → Edit
   - Account → Workers KV Storage → Edit
   - Account → Workers R2 Storage → Edit
   - Account → D1 → Edit
   - Account → Durable Objects → Edit
   - Account → Hyperdrive → Edit
   - Account → Queues → Edit
   - Account → Workflows → Edit
   - Account → Cloudflare Pages → Edit
   - Account → Access: Apps and Policies → Edit
   - Account → Turnstile → Edit
   - Account → Secrets Store → Edit
   - Account → Analytics Engine → Edit
   - Zone → Workers Routes → Edit (once you have a domain)
4. Account resources: your account.
5. Save the token. Also note your **Account ID** (top of the dashboard).

### 4.3 GitHub Personal Access Token (fine-grained)

1. https://github.com/settings/personal-access-tokens.
2. **Generate new token (fine-grained)**.
3. Scope: only the Boundless repo (you'll create it in step 7).
4. Permissions: Repository → Contents (read+write), Issues (read+write), Pull Requests (read+write).
5. Save.

### 4.4 APNs key (.p8) — Apple Push Notification service

1. https://developer.apple.com/account → **Certificates, Identifiers & Profiles** → **Keys** → **+**.
2. Name: `Boundless APNs`. Enable **Apple Push Notifications service (APNs)**.
3. Download the `.p8` file. **You can only download it once** — store it carefully.
4. Note the **Key ID** (10 chars) and your **Team ID** (10 chars, visible in account membership).

> The .p8 + Key ID + Team ID are what your Cloudflare Worker will use to sign APNs tokens. Store them in Cloudflare Secrets Store later.

### 4.5 FCM service account JSON — Firebase Cloud Messaging

1. https://console.firebase.google.com → create a project (use the same name as your Android app's package).
2. Project Settings → **Service accounts** → **Generate new private key**.
3. Download the JSON file. Store carefully.

## 5. Apple Developer applications

These have review lead times — start them on day one even if you're not coding for weeks.

### 5.1 Critical Alerts entitlement (required for the Doorbell notification)

The Doorbell notification ("Daniel is at your door") needs to wake the device even in Do Not Disturb. That requires the **Critical Alerts** entitlement, which Apple grants only for narrow use cases.

**Apply at:** https://developer.apple.com/contact/request/notifications-critical-alerts-entitlement

**What to write in the justification** (use your own words; here's the substance):

> Boundless is a transportation-coordination app for closed groups (families, congregations, communities) helping elderly and disabled members reach a recurring gathering. The Doorbell notification — sent only when a vetted, group-internal driver arrives at the door — must reach the rider reliably even when the device is in Do Not Disturb mode. Without Critical Alerts, an elderly rider who silences their phone overnight or during a service may miss the only notification of the day, leaving them stranded or causing the driver to wait outside. The notification is sent at most once per ride (driver arrival) and the rider can opt out at any time.

Expect 1–3 weeks for a decision. They may ask follow-up questions; respond promptly.

### 5.2 Push Notifications capability

Enabled per-target in Xcode. No application needed.

### 5.3 App Groups (for Widget extensions)

Enabled per-target in Xcode. No application needed.

### 5.4 (Optional) Sign in with Apple

If you decide to support this for member auth, enable it per-target. No external application needed.

### 7.5 Wire up your secrets

```bash

cp .mcp.json.template .mcp.json
```

```bash
grep -E '\.mcp\.json|\.claude/settings\.json' .gitignore
```

(They are; this command should print both lines.)

### 7.6 Install the git hooks

```bash

ln -s ../../.claude/hooks/pre-push.sh .git/hooks/pre-push
```

(The post-edit hook is wired through Claude Code's settings, not git.)

### 7.7 Sanity check


## 9. The daily loop

Once everything is wired, this is the rhythm:

```
┌──────────────────────────────────────────────────────────────────────┐
│  for each feature:                                                   │
│    1. /new-spec <name>      (you write the user-facing spec)         │
│    2. /clarify              (clarifier surfaces ambiguities)         │
│    3. /speckit.plan         (architect, via Spec Kit)                │
│    4. test-strategist       (designs the test plan)                  │
│    5. /speckit.tasks        (decompose into ordered tasks)           │
│    6. for each task:                                                 │
│       a. fresh session                                               │
│       b. docs-researcher for any new library                         │
│       c. implement (parent session, with hooks running on save)      │
│       d. /compact + end session                                      │
│    7. /ship-feature         (reviewer + security + i18n + parity)    │
│    8. open PR; CI runs; merge                                        │
└──────────────────────────────────────────────────────────────────────┘
```

### Habits that pay compounding interest

- **One session = one task.** End it. Start fresh.
- **Subagents do verbose work.** Your main context stays small.
- **ADRs > memory.** When you decide something, write it down.
- **Spec gates are not optional.** Skipping `/clarify` costs days later.
- **Test before declaring done.** No exceptions. The pre-commit hook is your friend.

### What to do when Claude gets it wrong

1. **Was the spec clear?** If no, that's the bug — go back and clarify.
2. **Was the right subagent invoked?** If you let Claude code in the main context what should have been a subagent task, you may have polluted context. End session, start fresh, delegate.
3. **Is a forbidden pattern triggering?** Check `docs/forbidden-patterns.md`. The reviewer should have caught it.
4. **Did the model hallucinate a library?** That's why `docs-researcher` is mandatory. Was it used?

## 10. Troubleshooting

### "MCP server failed to start"

- Check `.mcp.json` is valid JSON.
- Make sure Node.js 22+ is installed.
- For Brave: confirm your API key is active (test at https://brave.com/search/api/playground).
- For Cloudflare: confirm the package name `@cloudflare/mcp-server-cloudflare` is current — Cloudflare has shipped multiple MCP variants; check https://developers.cloudflare.com/agents/ for the canonical one.
- Restart Claude Code after editing `.mcp.json`.

### "Subagent not found"

- Confirm the file exists in `.claude/agents/`.
- Confirm the YAML frontmatter has both `name:` and `description:`.
- Confirm the file is `.md` not `.markdown`.
- Restart Claude Code.

### "Hook didn't run"

- Pre-commit / pre-push: `ls -l .git/hooks/pre-commit` — should be a symlink pointing to `../../.claude/hooks/pre-commit.sh`.
- Post-edit: configured via `.claude/settings.json` — verify JSON is valid and that the matcher fires (look for it under `hooks.PostToolUse`).

### "Spec Kit overwrote my constitution"

```bash
git checkout .specify/memory/constitution.md
```

### "I want to add another MCP server"

Edit `.mcp.json`, add the new server block, restart Claude Code. The pattern is:

```json
"server-name": {
  "command": "npx",
  "args": ["-y", "@org/package-name"],
  "env": { "KEY": "value" }
}
```

### "Claude is suggesting a library version that doesn't exist"

That's a hallucination. The protocol failed. Two fixes:
- Make sure `docs-researcher` is being invoked (its `description` should auto-route — if not, invoke explicitly).
- Update `docs/stack-matrix.md` with the actual version, so Claude reads it on next session.

### "I want to share this Claude Code setup with a collaborator"

`.claude/CLAUDE.md`, `.claude/agents/`, `.claude/commands/`, `.claude/hooks/`, `.specify/memory/constitution.md`, `docs/*`, and `.mcp.json.template` are all checked into git — your collaborator clones and runs through Section 7.5 onward.

`.mcp.json` and `.claude/settings.json` are per-developer (contain secrets / personal preferences). Each collaborator copies the templates separately.
