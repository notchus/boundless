## One-paragraph summary

The Rust admin Worker is already live on the edge (`/api/admin/members/*` + `/api/admin/audit-log`,
Hyperdrive â†’ Neon, cross-tenant isolation proven â€” spec 008 T11/AC16). Its other half, the SvelteKit
admin dashboard in `web/`, still only runs locally against in-memory fakes. This spec deploys that
dashboard to Cloudflare Workers and wires it end-to-end to the live Worker, so a real Admin (Sarah) can
open a Developer-issued onboarding link, register a passkey, sign in, and manage members against real
Neon Postgres â€” with her session, passkey, and invite persisted across cold starts. It does **not** add
any new member-facing behavior; it makes the admin surface a real, usable deployment instead of a local
dev preview.

## User story

As Sarah (Admin), I want the Boundless admin dashboard to be a real website I can sign into from my
laptop with my passkey, so that I can add members and manage the roster for our group without anyone
running a dev server for me.

## What changes for Maria? (rider primary persona)

**Nothing.** Maria never touches the admin web app. No rider-facing surface, string, or behavior changes.
This is admin-tier infrastructure only.

## What changes for Daniel? (driver primary persona)

**N/A.** No driver-facing surface changes.

## What changes for Sarah? (admin primary persona)

Everything that was previously "works on a local dev server" becomes "works at a real URL":

- She opens her Developer-issued onboarding link at the deployed admin domain and registers a passkey.
  Her credential is stored durably (not lost on the next request / cold start).
- She signs in with that passkey; her session survives navigation and Worker cold starts (persisted, not
  an in-memory map that resets).
- The member list, add-member, edit-member, regenerate-code, and audit-log screens show **real** data
  from the live Worker + Neon, instead of an in-memory fake.
- If the backend is misconfigured, she sees a calm error, never a fake roster silently served as if real.

**N/A.** Margaret/Tobias are riders/drivers; unaffected.

## Pinned decisions (resolved in `/clarify`, 2026-06-12)

These were the spec's open questions; they are now decided. The plan builds on them.

- **D1 â€” Persistence routing = Option B1.** The WebAuthn invite + credential persistence lives behind
  **new server-to-server admin-auth endpoints on the Rust Worker**, reached over the existing ADR-0026
  BFF shared secret. The web tier keeps **zero** direct Postgres access. The invite-token HMAC compare
  runs in the Rust **core** (`admin_invitation_token_matches`, ADR-0017 P4 carve-out); the WebAuthn
  ceremony verification and the TTL/consumed *verdict* (`evaluateInvite`) stay in edge-TS.
- **D2 â€” These new Worker endpoints are frozen contract.** They are added to `api/openapi.yaml` and pass
  the additive contract-freeze test, exactly as `/api/admin/members/*` was (spec 008). A new ADR records
  the B1 routing decision.
- **D3 â€” They are RLS-scoped by the Worker's `GROUP_ID` binding** (single-install), never a client value
  â€” identical to the member endpoints, inside a `SET LOCAL app.current_group_id` transaction. Invite
  resolve (which runs before any admin session) takes its group from `GROUP_ID`, not from the token.
- **D4 â€” No new migration.** Tables `0007_admin_webauthn_credentials` and `0008_admin_invitations` are
  sufficient; the credential CRUD and invite-consume map to their existing columns/indexes (incl. the
  `admin_invitations_one_live_per_admin` partial-unique index â€” the operator re-seed must respect its
  supersede-then-insert ordering).
- **D5 â€” Admin session backend = KV** (`ADMIN_SESSIONS`), not Postgres â€” under B1 a Postgres session
  would force yet another Worker endpoint. KV's revocation is best-effort within its propagation window;
  acceptable for a shorter-lived admin session, and documented as a known property (the same window the
  `CHALLENGES` consume-once path already lives with).
- **D6 â€” Two KV bindings:** keep `CHALLENGES`, add `ADMIN_SESSIONS`. `App.Platform` is generated by
  `wrangler types` (not the hand-written `app.d.ts`), and CI asserts the generated types match
  `wrangler.toml` so a binding rename fails the build.
- **D7 â€” Deploy domain / WebAuthn RP_ID.** The **intended production end state is a custom admin domain**
  (the user confirmed). For **dev/test only** the first deploy uses `*.workers.dev` (RP_ID =
  `boundless-admin-web.<account>.workers.dev`). `WEBAUTHN_RP_ID`/`WEBAUTHN_ORIGIN`/`WEBAUTHN_RP_NAME` must
  therefore stay **fully env-configurable** (no hard-coded host) so the custom-domain cutover is a config
  change, not a code change. Moving from the workers.dev host to the custom domain is a documented,
  expected **passkey-reset** event (one Developer re-invite per admin, survivable per ADR-0016 D4) â€” so
  any test admins enrolled on the workers.dev host are simply re-invited after the cutover. The runbook
  documents both the workers.dev (dev/test) and custom-domain (production) RP_ID values.
- **D8 â€” First-admin onboarding via an operator seed script** (not the Developer-minting UI, which stays
  deferred â€” spec 001 T08-shell). Two Workers stay two Workers (the ADR-0026 BFF boundary).

## Detailed behavior

This is a deployment + persistence-hardening slice. The dashboard's *screens* already exist (spec 001
T15 onboarding/sign-in; spec 008 T10 member management). What changes is what backs them.

### A. Member-management BFF â†’ live Worker

- `WorkerMembersClient` (already code-complete, `web/src/lib/server/members.ts`) is activated by setting
  the `ADMIN_WORKER_BASE` (the deployed Worker origin) and `ADMIN_API_SECRET` (the ADR-0026 BFF shared
  secret, **identical** to the Worker's `wrangler secret`) bindings. With both present in production the
  selector returns the real client; with either missing it **fails closed** (`Error('Admin member
  backend unavailableâ€¦')`) â€” it must never silently serve the in-memory fake in production.
- Every member request carries `authorization: Bearer <ADMIN_API_SECRET>` and `x-admin-id:
  <session-admin-id>` (the WebAuthn-verified Admin, the I5 audit actor) â€” already implemented; this spec
  proves it round-trips against the live Worker.

### B. Persistent admin session store

- Replace the in-memory `Map` in `web/src/lib/server/session.ts` with a durable store keyed by the
  opaque session id behind the existing httpOnly + Secure + SameSite=Strict cookie.
- Admin sessions are **separate and shorter-lived** than member sessions (ADR-0016): the store carries a
  TTL and supports rotation/explicit revoke (sign-out).
- Backend: a Cloudflare **KV** namespace (`ADMIN_SESSIONS`) with native TTL â€” the same binding pattern
  the WebAuthn `CHALLENGES` KV already uses; no new infra class, and (D5) no extra Worker endpoint (a
  Postgres session under B1 would need one). Sign-out deletes the KV key; revocation is therefore
  best-effort within KV's propagation window â€” acceptable for a shorter-lived admin session and recorded
  as a known property, not a bug.

### C. Persistent WebAuthn invite + credential stores (Option B1)

- Replace the in-memory `MemoryInviteStore` / `MemoryCredentialStore` (`web/src/lib/server/webauthn/
  testing/memory-stores.ts`) with durable persistence â€” but the web tier never touches Postgres. Per
  **D1**, the production `InviteStore` / `CredentialStore` ports become **Worker-backed adapters** that
  call new server-to-server admin-auth endpoints on the Rust Worker over the ADR-0026 BFF secret.
- **New Rust Worker endpoints (D2 â€” added to `api/openapi.yaml` and frozen):**
  - invite **resolve** (load the live invite for a token) and single-use **consume** (atomic
    `consumed_at` stamp on first successful registration);
  - credential **list active** (by admin), **insert**, **revoke-all-for-admin**, **bump sign-count**.
  - Exact paths/methods/shapes + the `adminSharedSecret` security scheme are fixed in the plan; they
    mirror the `/api/admin/members/*` request conventions (`authorization: Bearer <ADMIN_API_SECRET>`).
- **Split of responsibility (ADR-0017 P4 carve-out):**
  - the WebAuthn **ceremony verification** (`@simplewebauthn/server`) and the invite **TTL/consumed
    verdict** (`evaluateInvite`) stay in edge-TS;
  - the invite-token **HMAC compare** runs in the Rust **core** (`admin_invitation_token_matches`) â€” it
    is part of the Worker's invite-resolve, never recomputed in edge-TS.
- **I5 audit scope (D2/F2):** these endpoints disclose admin-credential/invite metadata, **not member
  PII**, so they are **not** `x-requires-audit`. The plan re-checks the contract-freeze coverage list to
  confirm none of the new operations expose a member-PII field (no silent regression of the I5 gate).
- **RLS (D3):** every new endpoint runs inside a `SET LOCAL app.current_group_id = <GROUP_ID>` transaction
  using the Worker's single-install `GROUP_ID` binding â€” never a client/token-supplied group. Invite
  resolve (pre-session) takes its group from `GROUP_ID`, not the token.
- **Migrations (D4):** none. Tables `0007`/`0008` are sufficient; the CRUD and consume map to existing
  columns/indexes, respecting the `admin_invitations_one_live_per_admin` supersede-then-insert ordering.

### D. First-admin bootstrap (dependency)

- A passkey onboarding requires both a **pending-admin `members` row** (role=`admin`, **no PII** â€” null
  name/address) and a single-use, TTL-bounded **`admin_invitations` row** that FKs to it. Minting these
  via Developer WebAuthn (spec 001 T08-shell) is **out of scope here**. This spec provides an **operator
  seed** script (analogous to `bootstrap-group.sh`, run as the DB owner under the Worker's `GROUP_ID`)
  that inserts both rows â€” preserving I11/ADR-0015 (only the Developer initiates admin access; the seed
  carries no PII and no credential material) and idempotent / non-overwriting against the
  `one_live_per_admin` index.
- **Known I5 gap (F5):** `admin_invitations.created_by` is meant to name the Developer who authorized the
  admin, but the seed has no Developer WebAuthn identity yet (deferred to T08-shell). The seed leaves
  `created_by` as a documented operator/system placeholder; this weakens the I11 audit-actor leg until
  the minting UI lands, and is recorded as a deferred item rather than silently ignored.
- **Schema check for the plan:** confirm `members` permits an `admin`-role row with null
  `name_encrypted`/`address_encrypted` (an admin has no rider/driver PII); if not, the plan resolves how
  a PII-free pending-admin is represented.

### E. `wrangler.toml` + deploy finalization

- **Two KV bindings (D6):** the real `CHALLENGES` id (replace the placeholder) + the new `ADMIN_SESSIONS`
  KV. The `ADMIN_WORKER_BASE`/`ADMIN_API_SECRET` secrets (the latter byte-identical to the Rust Worker's
  â€” D1). `WEBAUTHN_RP_ID`/`WEBAUTHN_ORIGIN`/`WEBAUTHN_RP_NAME` pinned to the **`*.workers.dev`** domain
  (D7). `send_metrics = false` (already set); only placeholder/`REPLACE_AT_DEPLOY_*` ids and **no** secret
  in `[vars]` (the credential-scan gate enforces this).
- `App.Platform` is generated by `wrangler types` (not the hand-written `app.d.ts`); CI asserts the
  committed generated types match `wrangler.toml`, so adding/renaming a binding (e.g. `ADMIN_SESSIONS`)
  without regenerating fails the build (D6).

### F. Strip dev-only seams from the production bundle

- The four `/api/test/{seed-invite,seed-member,seed-session,reset}` routes are `dev`-gated (â†’ 404 in
  prod). Confirm â€” and assert in CI â€” that `dev` inlines to `false` so they are tree-shaken / unreachable
  in the built artifact.
- **Don't break the existing e2e harness (F12).** Those seams currently seed/reset the in-memory stores
  the existing T15/T10 Playwright suites depend on. Before deletion, give them a replacement dev path
  against the durable backends (e.g. point the `dev`-gated reset/seed at the KV session store + the
  Worker-backed invite/credential adapters, or a small dev-only durable seeding helper) so the local e2e
  suites keep passing. The plan decides per-seam: delete vs. repoint-to-durable-store-in-dev.

### G. Web-tier scrubbed logging (P2/I10 backstop)

- Route uncaught endpoint throws (e.g. the prod-no-KV selector throws) through a web `emit()` scrubber
  sink + a `handleError` hook + a no-raw-`console` lint, with an I10 scrubber fixture for the operator
  strings. (Carry-forward from the spec 001 T15 register, items F1/F2.)
- **Invite-token-in-URL leak (F13).** The invite token rides in the URL path (`GET /api/admin/auth/
  invite/{token}`), and under B1 it now also transits **to the Worker** (invite-resolve). Both log paths
  must keep the `{token}` segment off the sink: the web `emit()` scrubber has a fixture for a URL-embedded
  opaque token (assert redacted), and the Worker invite-resolve endpoint keeps the presented token off its
  log path (the same I10 discipline the member error path already follows). Plus `Referrer-Policy:
  no-referrer` on the registration page (see AC11).

### H. Deploy runbook + live proof

- A `docs/runbooks/deploy-admin-web.md` mirroring `deploy-worker.md`: the human-gated `wrangler` steps
  (KV create Ă—2, secrets, deploy) in order, the WebAuthn domain config, the operator first-admin seed
  (Â§D), and the post-deploy smoke. It must pre-empt the two error-prone footguns the way `deploy-worker.md`
  pre-empts the KEK symmetry one: (a) `ADMIN_API_SECRET` on the web Worker **must be byte-identical** to
  the Rust Worker's, and can't be read back from Cloudflare â€” save it (a mismatch surfaces as the opaque
  `ADMIN_UNAUTHORIZED` 401); (b) **RP_ID is permanent** â€” changing the domain after passkeys exist
  invalidates them (D7).
- A scripted `smoke-deployed-admin-web.sh` (analogous to `smoke-deployed-edge.sh`) for the live proof:
  seed a Developer invite â†’ register a passkey â†’ sign in â†’ list/issue a member against the live Worker +
  Neon â†’ sign out and assert the post-sign-out cookie returns the `/admin/signin` redirect (session gone)
  â†’ assert `Referrer-Policy: no-referrer` on the invite route and that `RP_ID` is not `localhost`.

## States and transitions

Admin-facing states are unchanged from specs 001/008 (onboard â†’ register passkey â†’ signed-in â†’
member-management screens; calm error screens for expired/consumed invite, verification failure, backend
unavailable). This spec changes only the **durability** of the state behind them:

| State | Before (local/dev) | After (this spec) |
|---|---|---|
| Admin session | in-memory `Map`, lost on cold start | durable (KV/Postgres), TTL + revoke |
| Registered passkey | in-memory fake, lost on cold start | durable (`admin_webauthn_credentials`) |
| Pending invite | in-memory fake, dev-seeded | durable (`admin_invitations`), operator-seeded |
| Member roster | in-memory fake | live Worker â†’ Neon |
| Backend missing in prod | (n/a) | fail-closed error screen, never the fake |

## Edge cases

- **No driver available / opt-out / driver drops post-match / clock wrong:** N/A â€” no matching or
  rider/driver runtime surface in this spec.
- **Backend unreachable (Worker down / wrong secret):** the dashboard shows a calm "couldn't reach the
  admin backend" state and fails closed; it must never fall back to the in-memory fake in production
  (AC1). A wrong `ADMIN_API_SECRET` surfaces as the Worker's `ADMIN_UNAUTHORIZED` (401) â€” logged, not
  shown verbatim.
- **KV unavailable at runtime (no session/challenge binding in prod):** fail closed with a redirect to
  sign-in / a calm error, never a silent in-memory fallback (the selector already enforces this; extend
  to the session + invite/credential stores).
- **Cold start mid-session:** the session must survive (AC2); a registered passkey must survive (AC3).
- **Expired/consumed invite:** the two stable codes (AC4) surface the existing `admin.onboarding.
  invite_expired` copy.
- **Group key not bootstrapped on the Worker:** member reads fail closed as `ADMIN_GROUP_KEY_MISSING`
  (503) â€” the dashboard shows the calm `admin.member.group_key_missing` state; an operator runs the
  Group bootstrap (deploy-worker.md step 5b).
- **Dynamic Type / VoiceOver:** the admin web targets WCAG 2.2 AA keyboard-complete (unchanged from T10/
  T15); this spec must not regress it (AC12).

## Privacy notes

Which privacy invariants (I1â€“I12) does this touch? How are they preserved?

- **I11 (Admins issued only by the Developer) / ADR-0015** â€” the operator seed (D) inserts only a
  Developer-authorized pending-admin; there is still no signup/self-serve path. The seed carries the
  opaque invite token only (no PII, no credential material). **Known gap (F5):** `created_by` (the
  Developer audit actor) is a placeholder until the Developer-minting UI (T08-shell) lands â€” recorded as
  a deferred I5/I11 audit-actor gap, not silently dropped.
- **I5 (Admin PII reads audit-logged)** â€” preserved by the live Worker's existing `#[require_audit]` gate;
  the web BFF passes `x-admin-id` as the audit actor. The **new B1 admin-auth endpoints** disclose admin
  credential/invite metadata, **not member PII**, so they are not `x-requires-audit`; AC13 confirms the
  freeze-test coverage list still flags any member-PII disclosure. This spec adds no member-PII-returning
  path that bypasses the Worker.
- **I1 (addresses encrypted at rest)** â€” the web tier never holds the per-Group `GroupKey` or plaintext
  PII; all PII decryption stays in the Worker (ADR-0025). The dashboard only ever sees the Worker's
  already-projected wire DTOs.
- **I8 (no third-party trackers)** â€” the deploy must add no analytics/tracker; the network allow-list
  gate covers `web/`'s lock file.
- **I10 / P2 (no PII in logs)** â€” the web `emit()` scrubber sink (G/AC8) ensures operator strings and any
  inadvertent PII are redacted before persistence.
- **I3 / I9 / I2 / I12** â€” not touched by this slice (no phone-hash, live-tracker, matching, or deletion
  surface here).
